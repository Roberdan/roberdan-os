## Key decisions (with rationale)

- **Don't guess provenance from a filename** — the plan assumed `platforms/{chatgpt,codex,hermes}/README.md`
  were hand-written; a literal diff against the generator's heredoc output proved otherwise. Only
  `.github/copilot-instructions.md` actually is hand-written. Verify, don't pattern-match.
- **A verification-pass FAIL reuses the existing failure path** rather than inventing a parallel
  one — same `MAX_ATTEMPTS`/`attempts_file`/`escalate: true` machinery a task's own exit≠0 already
  uses. Less code, one failure semantics to reason about.
- **`kb start --by roberto` stays honor-system, on purpose** — a blocking identity check would
  break the documented autonomous "do all the todos" flow. An audit trail (visible, not
  enforced) was chosen over a false sense of security from an unenforceable check.
- **Salted hashes, not silence** — `test/leak-check.sh`'s tier (b) is an honest partial win: it
  stops casual reading of the denylist and keeps CI logs clean, but a committed salt cannot stop
  a dictionary attack against a guessed name. Documented as exactly that in three places
  (AGENTS.md, the script header, the CI workflow comment) rather than oversold.
- **Fix bugs a test surfaces, don't just report them** — two real, pre-existing portability/logic
  bugs (`sed -i ''` under GNU sed, the self-referential `mv` in factory/run.sh) were found while
  writing tests for this plan's own features, not the features being asked for. Both blocked
  `test/validate.sh` from being green, which the plan requires — fixed them in place, documented
  why in the commit messages.

## Honest scars (don't repeat)

**New this session:** a test that reaches a script's terminal state by calling it *twice* can
mask a crash on the very first call. The existing "failing task" factory test called
`bash factory/run.sh` twice to exhaust `MAX_ATTEMPTS=2` — which happened to work because the
first call's crash (the self-referential `mv` bug) still wrote the attempts-counter file before
dying, so the second call "accidentally" finished the job correctly. The bug was invisible until
a *single*-invocation test (added for the thor-verify PASS/FAIL cases) exercised the retry branch
on its own. Lesson: when a test wraps a script in a loop or repeats an invocation to reach a
desired end state, check what a *single* invocation does before trusting the pattern — the
repetition can hide exactly the crash you're supposed to be catching.

**Carried forward:** a safety check that isn't wired to run automatically is not a control — it's
a hope (prior session, re-affirmed: `test/leak-check.sh`'s tiers only help if something actually
runs them — CI tier (c) is silent by design when no denylist/hash file exists, which is correct
but easy to mistake for "checked and clean").
